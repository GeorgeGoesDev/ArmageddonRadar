# Armageddon Radar — Phase 3.6: APOD Polish

**Date:** 2026-07-17
**Status:** Approved design → ready for implementation plan
**Scope:** A bugfix plus a small feature on the existing APOD surface. Slots in
before **Phase 4b** (the Android home-screen widget), whose brainstorm is paused
mid-design and resumes after this ships.

## Goal

1. **Fix APOD staleness** — the banner shows yesterday's picture all day. Root
   cause diagnosed and confirmed on-device (see below).
2. **Fix midnight-resume staleness** — a latent defect in the same class,
   affecting APOD, the NEO feed, and the selected day.
3. **Add two APOD actions** — download the image to the gallery, and set it as
   the phone wallpaper.

Items 1–2 are pure JS. Item 3 adds three Expo modules and therefore a prebuild
and a rebuilt APK.

## Bug 1 — APOD shows yesterday's picture (confirmed)

### Evidence

- At 15:32 CEST on 2026-07-17, `api.nasa.gov/planetary/apod` returned
  `date: 2026-07-17` ("The Dust Trail of Comet Tempel 2").
- The device showed `2026-07-16` ("NGC 300: A Cosmic Gemstone with Stars and Gas
  Clouds").
- The stale image **survived a force-stop and cold start**, proving the bad entry
  lives in the *persisted* AsyncStorage query cache and is still considered fresh.

### Root cause

`useApod` keys the query on the **device's** calendar date, while `fetchApod`
sends no `date` param and therefore asks NASA for whatever is **currently
latest**:

```ts
queryKey: ['apod', getLocalDateKey()],           // Amsterdam's date
queryFn: ({ signal }) => fetchApod(apiKey, signal),  // NASA's "latest"
staleTime: ONE_DAY,
refetchOnWindowFocus: false,
```

NASA rolls APOD at **midnight US Eastern** — 06:00 in Amsterdam during CEST. So
between 00:00 and 06:00 local, NASA's "latest" is still *yesterday's* picture.
Any APOD fetch in that six-hour window writes yesterday's payload under **today's**
key. `staleTime: ONE_DAY` then pins it as fresh for 24 hours, and
`refetchOnWindowFocus: false` removes the only path that might have corrected it.
The entry is persisted, so it survives restarts. The result: yesterday's picture
for the whole day, resetting into the same trap at the next local midnight.

**The key changes on the device's clock; the resource changes on NASA's clock.
The gap between those two clocks is the bug.**

### Fix

Key the query on NASA's publication day, and refuse to pin a payload that isn't
the day we asked for.

```ts
// src/utils/dates.ts — pure
/**
 * The APOD publication day. NASA rolls APOD at midnight US Eastern; keying by the
 * device's date caches yesterday's payload under today's key for anyone east of
 * ET. Approximated as the date at UTC-5 (US Eastern standard) — deliberately
 * plain arithmetic, so it needs no ICU timezone data on Hermes.
 */
export function getApodDayKey(now: Date = new Date()): string {
  const shifted = new Date(now.getTime() - 5 * 60 * 60 * 1000);
  const y = shifted.getUTCFullYear();
  const m = String(shifted.getUTCMonth() + 1).padStart(2, '0');
  const d = String(shifted.getUTCDate()).padStart(2, '0');
  return `${y}-${m}-${d}`;
}
```

```ts
// src/utils/apodFreshness.ts — pure
export const APOD_FRESH_MS = 24 * 60 * 60 * 1000;
export const APOD_RECHECK_MS = 5 * 60 * 1000;

/**
 * Full-day caching only when the payload really is the requested day. NASA
 * sometimes publishes late; anything older must go stale fast so the next open
 * self-corrects instead of pinning a wrong picture for 24h.
 */
export function apodStaleTime(dataDate: string | undefined, apodDay: string): number {
  return dataDate === apodDay ? APOD_FRESH_MS : APOD_RECHECK_MS;
}
```

```ts
// src/hooks/useApod.ts — `useApodDayKey` is the foreground-aware hook from Bug 2 below
const apodDay = useApodDayKey();
return useQuery<Apod, Error>({
  queryKey: ['apod', apodDay],
  queryFn: ({ signal }) => fetchApod(apiKey, signal),
  staleTime: (query) => apodStaleTime(query.state.data?.date, apodDay),
  gcTime: ONE_DAY,
  retry: 1,
  refetchOnWindowFocus: false,
});
```

Functional `staleTime` (`StaleTime | ((query) => StaleTime)`) is supported by the
installed `@tanstack/react-query@5.101.2` — verified in
`node_modules/@tanstack/query-core` types.

**`queryFn` keeps requesting "latest" rather than sending `&date=`** — asking for a
day NASA has not published yet returns an error, whereas "latest + validate"
degrades gracefully.

**Why UTC-5 and not the true ET date:** the margin only reduces wasted fetches;
`apodStaleTime` is what guarantees correctness. UTC-5 rolls the key at 05:00 UTC —
an hour after NASA's EDT rollover (04:00 UTC) and level with the EST one
(05:00 UTC). A winter race is possible and self-heals within `APOD_RECHECK_MS`.

**This fix heals the currently broken install with no cache-buster bump.** Today's
key is still `2026-07-17`, so the poisoned entry remains — but its `data.date` is
`2026-07-16`, which fails the check, marks it stale, and refetches on the next app
open (`refetchOnMount` defaults to refetching stale queries).

**User-visible effect:** each day's picture appears from ~07:00 Amsterdam instead
of a stale one all day, always labelled with its true date (`ApodSheet` already
renders `apod.date`).

## Bug 2 — Date keys never roll while the app is resident

### Root cause

`getLocalDateKey()` is evaluated **during render**, and nothing in the app watches
the clock. A resident app resumed after midnight need not re-render, so the key
keeps yesterday's date:

- `useApod` — `queryKey: ['apod', getLocalDateKey()]`
- `useNeoWeek` — `queryKey: ['neo-week', startKey, apiKey]`
- `DashboardScreen` — `useState<string>(getLocalDateKey())` seeds `selectedDateKey`
  **once**, at mount.

This is a genuine latent defect, distinct from Bug 1 (the cold start ruled it out
as the current cause). The NEO feed is otherwise sound: NeoWs is keyed by calendar
date and never republished, so it has no equivalent publish-window problem.

### Fix

Re-evaluate the day key whenever the app returns to the foreground.

```ts
// src/hooks/useDayKey.ts
import { useEffect, useState } from 'react';
import { AppState } from 'react-native';
import { getApodDayKey, getLocalDateKey } from '../utils/dates';

/** Recomputes `compute` each time the app becomes active. `compute` must be a stable reference. */
function useForegroundValue(compute: () => string): string {
  const [value, setValue] = useState(compute);
  useEffect(() => {
    const sub = AppState.addEventListener('change', (state) => {
      if (state === 'active') setValue(compute());
    });
    return () => sub.remove();
  }, [compute]);
  return value;
}

export const useLocalDateKey = () => useForegroundValue(getLocalDateKey);
export const useApodDayKey = () => useForegroundValue(getApodDayKey);
```

- `useApod` uses `useApodDayKey()`; `useNeoWeek` uses `useLocalDateKey()`.
- `DashboardScreen` follows the rolling day:

```ts
const todayKey = useLocalDateKey();
useEffect(() => {
  setSelectedDateKey(todayKey);
  setSelectedId(null);
}, [todayKey]);
```

This runs harmlessly at mount (same value) and resets the selection only when the
day actually changes — matching what a fresh launch would show.

**Out of scope:** an app left *continuously active* across midnight still will not
roll (no timer). Resume is the realistic case; a ticking clock is YAGNI.

## Feature — APOD download + wallpaper buttons

A footer with two buttons, rendered **only when `apod.mediaType === 'image'`**
(hidden entirely on video days — no dead affordances). It sits inside the existing
`SafeAreaView edges={['bottom']}` in `ApodSheet`, pinned below the `ScrollView`, so
it clears the Android nav bar per the Phase 3.5 modal fix.

### Shared download (`src/utils/apodImage.ts`, side-effecting)

```ts
// Downloads the HD image (falling back to the standard url) into the cache dir.
export async function downloadApodImage(apod: Apod): Promise<File>
```

Uses SDK 57's `expo-file-system` class API: `new Directory(Paths.cache, 'apod')`,
`destination.create()`, `File.downloadFileAsync(url, destination)`.

### Download to gallery

```ts
export async function saveApodToGallery(apod: Apod): Promise<void>
```

1. `downloadApodImage(apod)`
2. `requestPermissionsAsync({ granularPermissions: ['photo'] })` — bail out with a
   message if denied.
3. `Asset.create(file.uri)`.

Both are named imports from `expo-media-library`
(`import { Asset, requestPermissionsAsync } from 'expo-media-library'`), not the
legacy default-namespace style.

**SDK 57 API note:** `saveToLibraryAsync` is deprecated and **throws at runtime**;
`Asset.create` is the current call. Confirmed against the v57 docs per `AGENTS.md`.

### Set as wallpaper

```ts
export async function setApodAsWallpaper(apod: Apod): Promise<void>
```

1. `downloadApodImage(apod)`
2. `IntentLauncher.startActivityAsync('android.intent.action.ATTACH_DATA', { data: file.contentUri, type: 'image/*', flags: 1 })`

`flags: 1` is `FLAG_GRANT_READ_URI_PERMISSION`, required for the target app to read
the URI. Android (MIUI's own wallpaper editor, on this device) then supplies crop
controls and the home/lock choice. No native code, no `SET_WALLPAPER` permission.

`File.contentUri` is the SDK 57 replacement for the deprecated
`FileSystem.getContentUriAsync()`. **Implementation must verify `contentUri` against
the installed `expo-file-system` types before use** — it is documented but unverified
locally, since the package is not yet installed.

### UI (`src/components/ApodActions.tsx`)

- Two `Pressable`s side by side: "Save to gallery" (`download` icon) and "Set as
  wallpaper" (`wallpaper` icon), styled with existing `colors` tokens and
  `MaterialCommunityIcons`, matching the sheet's existing buttons.
- Per-button state: `idle | working | done | error`. HD APODs run to several MB, so
  a spinner during download is required, and both buttons disable while either works.
- Failures surface inline as a short message under the buttons — never a silent
  no-op.
- Fires `hapticSuccess` on success, gated on `settings.hapticsEnabled` (existing
  pattern).

## Testing

Unit tests (`jest-expo`) for the pure logic:

- `getApodDayKey` — rolls at 05:00 UTC: `2026-07-17T04:59:59Z` → `2026-07-16`;
  `2026-07-17T05:00:00Z` → `2026-07-17`. A CEST-evening instant
  (`2026-07-17T21:00:00+02:00`) → `2026-07-17`. Independent of the device timezone.
- `apodStaleTime` — matching date → `APOD_FRESH_MS`; older date → `APOD_RECHECK_MS`;
  `undefined` data → `APOD_RECHECK_MS`.
- **Regression test for Bug 1** — the exact failure: data `2026-07-16` under key
  `2026-07-17` must yield `APOD_RECHECK_MS`, not a full day.
- `useDayKey` — with a mocked `AppState`, an `active` transition after the clock
  advances past midnight produces the new key; a non-`active` transition does not.

Native paths (download, gallery write, wallpaper intent) and the sheet footer are
verified with `tsc --noEmit`, `expo export --platform android`, and on-device.

## Acceptance criteria

- On next app open, the banner shows **today's** picture (2026-07-17, "The Dust
  Trail of Comet Tempel 2") without clearing app data — the poisoned entry
  self-heals.
- Opening the app between local midnight and ~07:00 shows yesterday's picture
  **labelled with yesterday's date**, and it rolls to today's without intervention.
- An app resumed after midnight shows the new day's feed, the new day's APOD, and
  the new day selected in `DaySelector`.
- On an image day, the APOD sheet shows two working buttons above the nav bar:
  "Save to gallery" writes the HD image to the gallery, and "Set as wallpaper"
  opens the system wallpaper flow with the image loaded. Permission denial and
  download failure both surface a message rather than failing silently.
- On a video day, neither button renders.
- All Phase 1–4a features continue to work; existing tests stay green.
